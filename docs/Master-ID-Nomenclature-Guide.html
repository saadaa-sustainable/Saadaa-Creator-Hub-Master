Anatomy of an ID

Every identifier is built on the creator's SIF. A deliverable adds a -P number; a collaboration adds a -C number. The two live in separate columns — a post_id never contains a -C, and a collab_id never contains a -P.

Read the parts

Worked on a real creator, SIF-1007, with two historic collabs (C1, C2 / P1, P2), onboarded again with 3 deliverables.
SIF-1007creator - P3deliverable

post_id — one per reel / static post. The P climbs per creator across every collab. Here P3 because P1–P2 are already used in history. No -C lives here.

SIF-1007creator - C3collab / order

collab_id — one per order. All deliverables of that order share it. C3 because the creator already had C1, C2 historically. The 3 new deliverables P3/P4/P5 all carry collab_id SIF-1007-C3.

When each ID is minted

The cardinal rule of the new project: reach-out creates the row but mints nothing. The Post (P) and Collab (C) IDs are both born at onboarding, the moment an order is attached.

STAGE 1

Reach Out

id (bigserial) ✓ SIF minted (new creator) post_id collab_id nomenclature

Row saved, shows “Pending”. Identified only by its internal id until onboarded.

STAGE 2

Onboarding (order)

post_id · P-block ✓ collab_id · C ✓ nomenclature ✓

mint_onboarding_block reserves the collab + the contiguous P-block in one locked call.

→
STAGE 3

Posting

post_link · post_date status → Posted

IDs unchanged. The deliverable gets its live link, date, and final status.
